PHASE B — SECOND-PASS AUDIT (2026-04-19, later the same day)
================================================================
After Tier 1-3 shipped, I re-walked the app end-to-end as if I were
a brand-new tenant clicking through the UI. Six additional defects
surfaced — all directly caused by the tightened backend not matching
the still-old frontend. Every one is now fixed.

[F1 BLOCKER] Browser fetches were cross-origin
  - lib/api-client.ts hit NEXT_PUBLIC_API_URL (backend:3001) with
    credentials: 'include'. HttpOnly cookie lives on the Next origin
    (app:5000), so it was never sent → 401 on every call.
  - Fix A: next.config.mjs → async rewrites({afterFiles:[/api/:path* →
    BACKEND_INTERNAL_URL]}). Next route handlers keep priority.
  - Fix B: lib/api-client.ts → same-origin fetch (empty base, 'same-origin'
    credentials) + global 428 auto-redirect to /change-password.

[F2 BLOCKER] No /change-password page existed
  - Backend middleware BlockUntilPasswordChanged() 428-locks every
    tenant-scoped call until must_change_password=FALSE, but the
    frontend had no UX to clear it. Users saw an empty dashboard.
  - Fix: new app/change-password/{page.tsx, change-password-form.tsx,
    actions.ts}. Server action forwards the HttpOnly cookie, parses
    the backend's new token from JSON, and re-sets the Next cookie.

[F3 HIGH] ChangePassword reused the stale JWT
  - Endpoint returned {success:true} but the caller's cookie still
    carried mcp=true — next request 428s again. User had to logout.
  - Fix: backend/handlers/auth.go#ChangePassword now mints a brand-new
    JWT with mcp=false and sets a fresh Set-Cookie atomically.

[F4 HIGH] Edge middleware ignored mcp claim
  - Users with mcp=true could navigate every dashboard page client-
    side; each page then called the API, got 428, and broke.
  - Fix: middleware.ts decodes SessionPayload.mcp, routes mcp=true →
    /change-password, and routes mcp=false trying to visit
    /change-password → '/'.

[F2 companion] login/actions.ts post-login branch
  - Added: read result.user.mustChangePassword from backend response;
    redirect to /change-password instead of the requested `redirectTo`.
    This avoids a flash of the dashboard shell before middleware kicks in.

[F5 MED] Tenants couldn't see SUPER_ADMIN shared AI providers
  - ScopedDB WHERE client_id=$tenant excluded NULL client_id rows,
    but those are the platform-wide system keys that every tenant
    should be able to USE (not edit).
  - Fix: new ScopedDBWithShared helper emits "client_id = $scope OR
    client_id IS NULL". Used by GetAIProviders. Mutation endpoints
    still use strict ScopedDB so shared rows stay read-only.
  - GetAIProvider / GetProviderModels now allow reading shared rows
    (previously 404'd them for tenants).

[F6 MED] GetAllModels leaked catalog to unauth'd principals (defense)
  - Fix: enforce EffectiveTenantScope at entry and drop the dead
    `SELECT ... WHERE is_active = true` that was never used.

[DX] One-click deploy assets
  - scripts/generate-env.sh: generates all required secrets with
    openssl, accepts --public-url and --evo-key flags, optionally
    writes to .env with timestamped backup of existing file.
  - DEPLOY.md: Coolify-first deploy walkthrough with troubleshooting
    table and post-deploy hardening checklist.

NEXT STEP FOR THE USER  (updated after Phase B)
================================================================
Full walkthrough in DEPLOY.md. Short version:

1. Clone + push this repo to any Git host Coolify can reach.
2. On the VPS (or locally with Docker):
     ./scripts/generate-env.sh --public-url=https://app.your-domain.com --write
   (generates all five REQUIRED secrets with openssl and writes .env)
3. Docker path:
     docker compose up -d --build
     docker compose logs -f backend
   Coolify path:
     New Resource → Docker Compose → paste the .env you just generated
     → attach domain → Deploy
4. Browse to https://<your-domain>/login
     email:    admin@mantra.ai
     password: MantraAdmin2024!
   You will be forced through /change-password before any dashboard
   endpoint responds — by design.
5. Install scripts/backup-postgres.sh in a daily cron on the VPS host
   (see scripts/README.md).
